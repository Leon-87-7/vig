# vig — Domain Context

This document is the single source of truth for domain language and architecture decisions in this repository. It grows lazily via `/grill-with-docs` sessions as new terms and decisions crystallise.

---

## Glossary

| Term                             | Definition                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Job**                          | A unit of work representing one URL submitted by a user. Lives in the `jobs` table. ID format: `YYYYMMDD_HHMMSS_XXXX`.                                                                                                                                                                                                                                                                                                                           |
| **Shared job-creation core**     | `create_and_enqueue_job(chat_id, url, content_type, *, template=None, message_id=None) -> job_id` in `src/services/jobs.py` (ADR-0033). Owns dedup (`database.find_recent_job_by_url`, returning the existing job on a cache hit instead of duplicating) and the create+enqueue write. Three callers: the Telegram webhook, the dashboard's `POST /api/jobs` ([[Web dashboard]]), and the [[Repo follow-up]]. Each caller owns its own result notification (Telegram message, JSON response, or silent skip) — the core does not.                                                                                                                                                                                                                          |
| **Short video**                  | Instagram Reel, TikTok, or YouTube Short — processed via `processors/short_video.py`. Frames → Gemini Vision → links → Brave Search. A transcript is **always** produced as a best-effort tail (DB + Drive + Telegram document), independent of any prompt template — see [[Short transcript step]] and ADR-0020. Gemini text enrichment runs only when a template is set.                                                                                                                                                                                                                                                                                        |
| **Long video**                   | Full-length YouTube video — processed via `processors/long_video.py`. Two-phase pipeline: Phase 1 produces a transcript markdown uploaded to Drive; Phase 2 is Gemini enrichment, triggered either by user confirmation (plain-URL jobs) or automatically (explicit-command jobs). Unlike short video, enrichment is a separate queued task, not inline.                                                                                           |
| **Content type**                 | Enum: `short`, `long`, `article`, `repo`, `document`, or `link`. All but [[Link pipeline]] are detected at webhook routing time by `validators.detect_pipeline`; `link` is explicit-only (see [[Link pipeline]]).                                                                                                                                                                                                                                                                                                                                           |
| **Feed tab**                     | Top-level inventory switcher on the dashboard feed. The **Feed** tab shows every job as a list of rows (current behavior); each typed job tab (`short`/`long`/`article`/`repo`) shows only that job type as a grid of [[Preview card]]s; the **Links** tab shows the URL inventory. Feed search composes *within* the active tab: for job tabs it searches job fields, and for Links it searches link inventory fields (`url`, `title`, `description`, plus exact tag names — never `topic`, see [[Standalone link identity]]). Status filters apply only to job tabs and are hidden on Links because link inventory does not share the job status lifecycle. One tab is active at a time and is reflected in the URL (`?type=…` for job tabs, equivalent view state for Links). Docs ingest is an action/modal, not a Feed tab.                                                                                                                                                                                                                                                                                        |
| **Preview card**                 | Grid-cell representation of a job inside a typed [[Feed tab]] (vs the list row used in **Feed**). Shows the [[Thumbnail]], title, status badge, and timestamp. Landscape (16:9) for long/article/repo; portrait (9:16) for short. The redundant type badge is dropped (the tab implies the type); short cards keep a platform glyph since the tab mixes IG/TikTok/Shorts.                                                                                                                                                                                                                                                                                  |
| **Thumbnail**                    | The per-job preview image, resolved server-side into a single `thumbnail_url` (+ `thumbnail_kind`: `landscape`/`portrait`/`null`) on the `/api/jobs` payload — never derived client-side. Resolution by content type: long & YouTube-Shorts → `img.youtube.com/vi/<id>/…` (URL-derived); repo → `opengraph.githubassets.com/<any-hash>/<owner>/<repo>` (URL-derived); IG/TikTok short → stored [[Best frame]] served via owned `/api/jobs/{id}/thumbnail`; article → scraped `og:image` URL; none → `null` (typed placeholder). See ADR-0025.                                                                                                                                                                                                                                                                                  |
| **Best frame**                   | The single most representative frame of a short video. The index is chosen by Gemini Vision (`main_frame_index`) during processing; the frame bytes are otherwise ephemeral (from the [[Frame service]]). For IG/TikTok shorts the bytes are persisted as the job [[Thumbnail]] (ADR-0025). Backfill without Gemini re-picks via a local sharpness heuristic. YouTube Shorts skip this entirely — their thumbnail is URL-derived.                                                                                                                                                                                                                                                                                  |
| **Status FSM**                   | Job status lifecycle: `pending → processing → transcript_done → enriching → done` (or `error`/`cancelled`).                                                                                                                                                                                                                                                                                                                                      |
| **Enrichment**                   | Gemini text pass that classifies the job: category, topic, objective, action points, tools, market data.                                                                                                                                                                                                                                                                                                                                         |
| **Mini-PRD**                     | AI-generated product requirements document derived from a long-video transcript. Two slots: `auto` (no user input) and `intent` (user-supplied direction text).                                                                                                                                                                                                                                                                                  |
| **PRD auto slot**                | Worker-owned lock (`prd_auto_status='generating'`). Single acquirer — only the worker may set this.                                                                                                                                                                                                                                                                                                                                              |
| **PRD intent slot**              | User-triggered PRD with a direction text up to 1000 chars. Arms a `chat_state` row with a 10-minute `awaiting_intent` window.                                                                                                                                                                                                                                                                                                                    |
| **Task envelope**                | JSON dict pushed to the Redis queue: `{"task": <discriminator>, "job_id": <id>}`. Optional flags: `skip_document: true` (article retry — suppresses the `.md` Telegram document re-send).                                                                                                                                                                                                                                                        |
| **Recovery panel**               | Tenant-scoped dashboard controls that let a user reconcile only their own stuck or failed jobs.                                                                                                                                                                                                                                                                                                                                                  |
| **Clear failed**                 | Recovery action that marks owned `error` jobs as `cancelled`; it never hard-deletes job rows.                                                                                                                                                                                                                                                                                                                                                   |
| **Second Brain**                 | Semantic link graph: every URL extracted from any pipeline is embedded (768-dim via Gemini) and stored in `links` table + uploaded as Obsidian `.md` to Google Drive.                                                                                                                                                                                                                                                                            |
| **Brain ingest**                 | Fire-and-forget `brain.ingest_links(links, topic, source_job_id)` — does not block the user-facing response.                                                                                                                                                                                                                                                                                                                                     |
| **Brain graph**                  | The Brain-page visualization of the [[Second Brain]] as nodes + edges. Desktop renders a 2D force layout (`react-force-graph` `ForceGraph2D`); mobile renders an ego-network (one node + its neighbors). Served by `GET /api/brain/graph`. See ADR-0028.                                                                                                                                                                                          |
| **Graph node**                   | One `links` row, keyed by its **normalized URL** (one-node-per-URL). Carries `title`, `topic`, `seen_count`, embedding, and — for [[Repo node]]s — refreshable `stars`/`pushed_at`.                                                                                                                                                                                                                                                              |
| **Graph edge**                   | A derived semantic connection: a top-k cosine-similarity neighbor (≥ `BRAIN_MIN_SCORE`) computed from embeddings when the graph is built. **Not persisted** — there is no edges table. Reuses `brain._compute_related`.                                                                                                                                                                                                                          |
| **Topic cluster**                | A visual grouping of nodes sharing a `topic`, expressed as node color. There is no clustering algorithm and no dimensionality reduction — `topic` _is_ the cluster. See ADR-0028.                                                                                                                                                                                                                                                                |
| **Link tag**                     | A tag attached to a `links` row (`link_tags`: `link_id ↔ tag_id`, FK to `links.id`), answering "what is this URL". **Coexists** with job tags (`job_tags`, "what is this job/run about") — the two may disagree, no reconciliation. Reuses the per-chat `tags` vocabulary. Private to their owner in every view, including the [[Community Brain]]. Resolved 2026-07-11 (task 11 grill, session 1; timer/UI details still open — see `docs/TASK.md` task 11). **UI (resolved 2026-07-17, task 30 grill):** rendered as name-less **color dots + optional icons** (nullable `tags.icon`); the cluster itself opens the shared `TagMenu` picker; tag palette is a global ~10-hue OKLCH set excluding the signal-orange→pending-yellow band (replaces `PRESET_COLORS`, shared with job tags); the Links search box matches tag names **exactly** (join), OR-ed with substring url/title/description.                                                                                                                                                                        |
| **Standalone link identity**     | A Brain link's title + description come from the **URL itself**, never from the video it was extracted from (resolved 2026-07-17, task 32 grill). Sourcing is tiered: GitHub service → plain HTTP meta-tag parse → [[Jina Reader]] when vague/<40 chars. New `links.description` column; `links.topic` is demoted to **provenance only** ("seen in a video about X") — excluded from search, the embedding doc (`url+title+description`), and the table row; it surfaces only as the last line of the expanded More panel. Backfill rides the Brain refresh loop (`description IS NULL` = repair condition, re-embeds). Fixes sibling leakage: searching "svg" no longer returns every link extracted alongside an svg video. See `docs/TASK.md` task 32.                                                                                                                                                                        |
| **Link row**                     | The collapsed representation of one `links` row in the Links [[Feed tab]] — one format on both viewports (mobile card and desktop table): favicon → [[Trimmed URL]] → open-in-new-tab button → [[Link tag]] cluster. The favicon reuses the typed-tab `PlatformGlyph`, widened so any unknown host tries its real favicon (brand glyph for known platforms → favicon service → document-icon fallback). Favicon + trimmed URL form one anchor opening the link in a new tab; the external-link icon is a second, visually separate button to the same URL. The full URL never renders in the collapsed row: it lives in the anchor's tooltip (hover surfaces) and as the first line of the expanded More panel — above title · description, above provenance — so **every** row renders a More button, including rows with no title/description (previously those had none, leaving touch users no path to the full URL). Resolved 2026-07-19 (url-button-layout grill; fixes long URLs shoving the tag button off the card). |
| **Trimmed URL**                  | The row-identity text of a [[Link row]]: pathname + query string, with scheme and hostname dropped — the favicon carries the domain. When the path is empty or a bare `/`, the hostname is shown instead (a favicon alone can't identify an unknown domain). Deliberately different from the Telegram [[`/find` command]] display (netloc + path): Telegram has no favicon column, so it keeps the netloc. |
| **Community Brain**              | Opt-in shared tab on the Brain page: the pool of links stamped shared at ingest by users in an active [[Sharer window]], merged by canonical URL, viewable by everyone. Contrast with the per-tenant [[Second Brain]] (each user's own links; `links` gains `chat_id`, dedup key `(chat_id, url)`). Sharing is forward-only: opting in never shares the back-catalog, opting out never withdraws already-shared links.                                                                                                                                                                        |
| **Sharer window**                | The timed state (~8–12h) during which a user is a sharer: links they ingest are stamped shared into the [[Community Brain]] at ingest time. Stored per-user (user-settings store). Exact expiry semantics (fixed/silent vs. notice vs. sliding) unresolved — task 11's next grill session.                                                                                                                                                        |
| **Repo node**                    | A [[Graph node]] whose normalized URL is a GitHub repo (distinguished by URL host only — no type column). A scheduled job refreshes its `stars`/`pushed_at` **metadata only** (no Gemini re-analysis) when older than 14 days and not archived. See ADR-0027.                                                                                                                                                                                    |
| **Photo pipeline**               | Inline (no DB job, no queue) path for Telegram photo messages. Gemini Vision extracts verbatim-grounded URLs; `_filter_grounded_links` drops hallucinations.                                                                                                                                                                                                                                                                                     |
| **Verbatim**                     | The exact phrase Gemini quotes from the image that proves a URL/domain is literally rendered as text (not inferred from a brand name). Required field for photo link extraction.                                                                                                                                                                                                                                                                 |
| **UI chrome**                    | Social media interface elements (follower counts, "Followed by X", timestamps) that are not promoted resources. Photo pipeline drops links whose verbatim matches any pattern in `_UI_CHROME_PATTERNS` (includes "followed by" and related account-context phrases). Fixed to cover all UI-chrome variants in commit 2df529e (PR #48).                                                                                                           |
| **Transcript service**           | Python sidecar (`vig-transcript` container) that downloads videos via yt-dlp and returns transcript text. On caption-less videos, falls back to returning raw audio bytes (base64) for the worker to send to Gemini. Called by short/long processors via HTTP.                                                                                                                                                                                   |
| **Audio fallback**               | When the transcript service finds no VTT caption files, it downloads audio-only via yt-dlp and returns `{"audio_b64": "...", "mime_type": "audio/...", "fallback": "audio"}`. The worker, not the transcript service, sends this to Gemini.                                                                                                                                                                                                      |
| **Audio enrichment**             | A single Gemini `generate_content` call that receives inline base64 audio + the enrichment/template prompt. Gemini transcribes and analyzes in one shot, and **returns the transcript text alongside the analysis** (ADR-0020) so a caption-less template job persists its transcript without a second call. Used in the template path when captions are unavailable. Preserves the two-call budget (Vision + Audio-Enrichment). For a caption-less job with **no** template, a transcription-only Gemini call ([[Short transcript step]]) is made instead.                                                                                                                                                                |
| **Repo enrichment**              | Post-processing step in the photo pipeline that fetches GitHub API metadata (stars, forks, last-push date, primary language) for each `github.com` link extracted by Gemini Vision, then sorts the list by stars+forks descending before delivery. Non-GitHub links pass through unenriched.                                                                                                                                                     |
| **Enrichment sort**              | The ordering applied after repo enrichment: primary sort key is stars descending, secondary is forks descending. Applied only when enrichment succeeds for at least one repo.                                                                                                                                                                                                                                                                    |
| **Repo enrichment fallback**     | If the GitHub API calls fail (network error, rate limit, all 404s), the photo pipeline falls back to the plain extracted list in original order — current behavior, no metadata.                                                                                                                                                                                                                                                                 |
| **GitHub token**                 | Personal access token (or GitHub App token) stored as `GITHUB_TOKEN` in settings. Required for repo enrichment — if absent, enrichment is skipped and the plain list is returned. Authenticates REST/GraphQL calls; raises the rate limit from 60 to 5,000 requests/hour.                                                                                                                                                                        |
| **Repo metadata cache**          | Redis keys of the form `github_meta:{owner}/{repo}` storing GitHub API metadata as JSON. TTL: 24 hours. Checked before every API call; a cache hit skips the network round-trip entirely.                                                                                                                                                                                                                                                        |
| **Repo enrichment API strategy** | GitHub REST API v3 (`GET /repos/{owner}/{repo}`), one request per repo, all fired concurrently via `asyncio.gather`. Simpler than GraphQL; parallel execution keeps wall time acceptable even on cold cache.                                                                                                                                                                                                                                     |
| **Enriched repo line format**    | Four metadata fields per repo: ⭐ star count, 🔀 fork count, 💻 primary language, 📅 human-readable last-push age (e.g. "3 days ago", "8 months ago"). Open-issues count and activity-status labels ([INACTIVE] etc.) are excluded.                                                                                                                                                                                                              |
| **Enriched repo description**    | For repos that enrich successfully, the GitHub API `description` field replaces the Gemini-generated label. For repos that fail enrichment (404, timeout), the Gemini label is kept as-is.                                                                                                                                                                                                                                                       |
| **Repo enrichment scope**        | Only `github.com` URLs are enriched. Non-GitHub links (GitLab, direct sites, social handles) pass through unenriched in original position. Inline filter buttons (language/activity filters via Telegram keyboard) are deferred to a future issue.                                                                                                                                                                                               |
| **Template path fork**           | Caption-based Reels/YouTube follow the text transcript → enrichment path. Caption-less Reels on a template job follow the audio bytes → audio enrichment path. Both paths produce the same enrichment output shape. Transcript acquisition is now **decoupled** from this fork — see [[Short transcript step]]; the fork governs enrichment only.                                                                                                                                                                                                                              |
| **Short transcript step**        | The guaranteed best-effort transcript tail of the short pipeline (ADR-0020). Runs on **every** short job regardless of prompt template. Fetches once up front (captions are free; caption-less auto-routes to a transcription-only Gemini call via the free→paid key, or rides the fused [[Audio enrichment]] call on template jobs to stay within the two-call budget). Persists `jobs.transcript` + runs `key_phrases`, then — always as the pipeline's **last** emission, after any enrichment — uploads `{job_id}_transcript.md` to Drive and sends it as its own Telegram document (separate from the `{job_id}_short.md` analysis artifact). Non-fatal: vision already marked the job `done`. Explicit failure notes: sidecar error → `⚠️ Transcript service error: <msg>`; Gemini transcription fail → `⚠️ Transcription failed — Gemini unavailable`; succeeds-but-empty → `⚠️ I'm wordless`. Guarantees the transcript *attempt*, not a transcript in 100% of cases. |
| **Frame service**                | HTTP endpoint that extracts key frames from a video URL. Used by the long video pipeline.                                                                                                                                                                                                                                                                                                                                                        |
| **Brain refresh**                | Cron job (Sun + Wed 09:00 UTC) that re-embeds `links` rows whose `updated_at` is stale.                                                                                                                                                                                                                                                                                                                                                          |
| **Reaper**                       | Boot-time cleanup coroutine(s) run in `worker.loop()`. Three reapers: `prd.reaper()` / `prd.reaper_intent()` reset stuck `generating` PRD locks; `reap_stale_jobs()` resets orphaned `processing`/`enriching` jobs (older than 10 min) to `error`, bumps `attempt`, and notifies the user. None re-queue — recovery is mark-error + manual retry (ADR-0010).                                                                                     |
| **Orphaned job**                 | A job left in `processing` or `enriching` by a worker crash — a consequence of the at-most-once Redis queue (ADR-0002). Recovered at the next worker startup by `reap_stale_jobs()`: reset to `error`, `attempt` incremented, user notified (enrichment → retry button; video → resend link). Never silently re-queued. See ADR-0010.                                                                                                            |
| **Batch mode**                   | Photo pipeline feature: when the user sends multiple photos from the gallery as a single selection, Telegram delivers them as a media group — each message shares the same `media_group_id`. The bot accumulates file IDs per group in Redis (`photo_group_files:{media_group_id}`), using a 1-second in-memory debounce timer per group held in `_BATCH_TASKS` in `webhook.py`. When the timer fires, all images are sent to Gemini in one `call_gemini_photo_links` call and a single unified result is returned. Photos without a `media_group_id` go through `_handle_single_photo` unchanged. No explicit commands needed. See ADR-0024.                                                                                                                                                                                                                                                                                |
| **Slice**                        | Implementation milestone from the PRD (e.g. "slice #6 = Mini-PRD auto slot"). Slice numbers appear in code comments as a navigation aid.                                                                                                                                                                                                                                                                                                         |
| **Webhook dispatch table**       | Two module-level dicts in `src/telegram/webhook.py` mapping discriminator keys to handler functions: `_CALLBACK_TABLE` (callback-data prefix before `:` → `_cb_*` handler) and `_SLASH_TABLE` (command string → `_cmd_*` handler, assembled at import by merging hardcoded entries with the template commands generated from `PROMPT_TEMPLATES`). The router parses the key, looks up the handler, builds a context object, and calls it — no inline conditionals. A multi-file split (extracting `dispatch.py`/`callbacks.py`/`domain_cmds.py`/`photo.py`) was proposed and closed **wontfix** — `src/telegram/` stays two files (`webhook.py`, `sender.py`); webhook complexity is addressed by targeted CC-reduction of helpers within the single file instead. |
| **Telegram sender seam**         | `src/telegram/sender.py` is the single adapter to the Telegram Bot API and the canonical test seam. Modules in the `src/telegram/` package call it module-qualified (`sender.send_message(...)`, not a bare imported name) so tests patch `src.telegram.sender.*` once — independent of which module owns a given handler. Scope is the telegram package only; `worker.py` and the processors keep their `from src.telegram.sender import …` style and their own patch targets. See ADR-0015. |
| **CallbackCtx**                  | Context dataclass defined directly in `src/telegram/webhook.py` (see [[Webhook dispatch table]]) passed to every callback handler: `chat_id`, `job_id` (payload after `:`), `cq_id`, `data` (full raw string). Callback handlers never parse `data` themselves.                                                                                                                                                                                                                          |
| **SlashCtx**                     | Context dataclass defined directly in `src/telegram/webhook.py` (see [[Webhook dispatch table]]) passed to every slash command handler: `chat_id`, `parts` (split command + args), `message_id`. Does not carry `cq_id` — slash commands have no callback query to acknowledge.                                                                                                                                                                                                          |
| **Cancel exemption**             | `/cancel` is the only slash command exempt from the router's pre-clear behavior (clear `chat_state` + delete `pending_template` key). It reads `chat_state` itself before clearing, so it must own the clear internally. All other slash handlers run after the router has already cleared state.                                                                                                                                                |
| **PRD skeleton**                 | Single `run_prd(job, *, lock_col, model, build_prompt)` function in `prd.py` that owns the shared 7-step PRD pipeline: lock acquire → transcript sample → Gemini generate → Drive upload-or-update → Sheets append → brain ingest → Telegram delivery (+ error handling). `run_auto` and `run_intent` are thin wrappers that construct the right arguments and delegate. `PRD_JSON_SCHEMA` is a constant shared by both slots — not a parameter. |
| **GeminiClient**                 | Thin backward-compat wrapper (`src/services/gemini_client.py`) that re-exports the unified `src/services/gemini.py` module. The canonical module owns the single free→paid key fallback loop (`_call_with_fallback`), one `_call_sync`, one `_extract_json`, and four public wrappers: `generate` (text), `call_gemini_vision` (video frames), `call_gemini_photo_links` (photos), `resolve_tool_urls` (URL resolution). Vision and embeddings no longer keep their own loops — see ADR-0011. |
| **GeminiUnavailableError**       | Exception raised from `src/services/gemini._call_with_fallback` when both Gemini keys fail. Canonical for all call paths (text, vision, photo, embed) — supersedes the pre-ADR-0011 split where vision/photo raised `RuntimeError`. Callers translate it into their domain error (e.g. `EnrichmentUnavailableError`) or handle it directly.                                                                                                      |
| **Promise gap**                  | Enrichment output field that identifies the gap between what a video's title/thumbnail promises and what the content actually delivers. Extracted by Gemini during enrichment, persisted in `jobs.promise_gap`, and rendered in the Telegram delivery message.                                                                                                                                                                                    |
| **Prompt template**              | A named enrichment mode (`summary`, `method`, `technical`, etc.) that injects extra instructions into the Gemini prompt and optionally appends a structured `template_analysis` block to the enrichment output. Selected explicitly by the user via a slash command (e.g. `/method <url>`) or auto-detected from the video's title/transcript keywords.                                                                                           |
| **Template auto-routing**        | Automatic template selection at job-creation time: `detect_template` scores the video title + description against each template's `trigger_patterns` and picks the highest-scoring match, defaulting to `summary` on a tie or no match. Users can override with an explicit slash command; a mismatch warning is shown when the explicit choice conflicts with auto-routing.                                                                       |
| **Template analysis**            | A structured JSON sub-object appended by Gemini to the enrichment output for non-`summary` templates (e.g. `steps`/`common_mistakes`/`pro_tips` for `method`). Stored in `jobs.template_analysis` and rendered alongside the standard enrichment fields. `summary` jobs never produce a `template_analysis` block.                                                                                                                               |
| **URL deduplication**            | Per-chat gate that blocks resubmission of a URL whose job is still pending, processing, or done. Bypassed when a template slash command is the active trigger (explicit reprocess intent). The duplicate notice includes a "Show job done" inline button that forwards the original completion message. **Content-type-agnostic**: the check (`find_recent_job_by_url`) matches on `chat_id + url` only, so a dedup hit can land against a job of a *different* content_type than the one being requested — most callers treat this as an ordinary cache hit, but see [[Link pipeline]] for the one caller required to surface it as a hard warning instead.                                                                                                                                            |
| **Link pipeline**                | Sixth [[Content type]] (`link`): direct-add-to-[[Second Brain]], no processing/enrichment. Explicit-only — created via Telegram `/addlink <url>` or the dashboard's `U`-triggered "Add Link" modal, never via `validators.detect_pipeline` auto-routing, so a Reel/repo/video URL added this way becomes a plain `link` job, not a `short`/`repo`/etc. one. The processor fetches the page, extracts the [[Essential OG collection]], marks the job `done` (FSM: `pending → processing → done`, no intermediate states), then calls `brain.ingest_links` with its own real `source_job_id` — no fabricated job row, no nullable-FK special case in `brain.py`. Unlike its five sibling pipelines, it does **not** write a Sheets row or a separate Drive export; the Brain ingest's own `links` row + Obsidian `.md` upload is the only artifact. Both entry points carry an explicit warning that this is not the paste-a-URL / `N` pipeline-detection flow. See ADR-0039.                                                                                                                                            |
| **Essential OG collection**      | The harvested tag set for a [[Link pipeline]] job: `og:title`, `og:description`, `og:site_name`, `og:type`, `og:image`, `twitter:card`, `twitter:site` — extracted by an expanded `src/utils/og_image.py` (previously `og:image`-only). Flattened into one string and stored as that link's `links.topic` value, surfacing as "From: …" in the Links table — deliberately the raw harvested tags, not a content summary or a generic "added manually" note, since exposing what was scraped *is* this pipeline's output. `og:image` is also written directly onto `links.og_image_url` at ingest time (a new optional field on `ingest_links`' link dict) so the Links-table preview has it immediately instead of waiting on [[Brain ingest]]'s usual lazy on-demand fetch.                                                                                                                                            |
| **Force reprocess**              | `/force <url>` bypasses the dedup gate and resets the existing job row in-place — same job ID, `attempt` incremented, all result fields cleared — before re-enqueuing. Falls back to creating a fresh job only when no prior row exists for that URL in the chat.                                                                                                                                                                                 |
| **Ignored domain**               | A user-managed per-chat blocklist stored in the `ignored_domains` SQLite table. Added via `/ignore <domain\|URL>` (space-separated, one or more per call), removed via `/unignore`, listed via `/ignore_list`. `github.com` is protected and cannot be added. The short pipeline's `filter_vision_links` receives the list as `extra_ignored` and drops matching links before Brave enrichment.                                                   |
| **Enrichment confirmation gate** | Inline keyboard sent to the user after a long-video transcript is ready, offering "👎 No Thanks / ✨ Run Gemini / 📐 Build Spec". Present only for plain-URL long-video jobs (`template_detection_method != "explicit_command"`). Skipped entirely when the job was submitted via a template slash command — the worker auto-enqueues the enrichment task without asking. Tapping "✨ Run Gemini" now opens the **template picker keyboard** instead of immediately enqueuing enrichment. |
| **Template picker keyboard**     | Sub-keyboard shown after the user taps "✨ Run Gemini" in the enrichment confirmation gate. Presents all five prompt templates (summary, method, technical, review, narrative) plus "✍️ Freestyle" as inline buttons (3×2 layout). After the user picks any option the keyboard collapses to "You chose {template}" and the chosen analysis is enqueued. Picking "✍️ Freestyle" arms an `awaiting_freestyle` chat state instead of immediately enqueuing. |
| **Freestyle prompt**             | A user-supplied Gemini instruction that replaces the standard template's `extra_instructions`. The user types any free-form text; the enrichment worker substitutes it in place of the template's structured extraction instructions. Available in both pipelines: as a button inside the **template picker keyboard** (long video) and as a `/freestyle <url>` slash command (both pipelines). | **Feed freshness model**          | How the [[web-dashboard]] feed stays current without page reloads. The feed fetches the [[Tenant]]'s **full** job list once and does all content-type/status/search filtering **client-side** (extending the fuse.js-search precedent) — so pressing a filter is instant and never re-hits the API. Every job is born **out-of-band** via Telegram (the dashboard never initiates one), so *discovery* is the primary concern: freshness comes from three **silent** (no-skeleton, no scroll-jump) refetch triggers — (1) a 10 s **in-flight poll** while any job is `pending`/`processing`/`enriching`, (2) a ~2 min **backstop poll** for out-of-band arrivals while idly browsing, (3) **refetch-on-focus** when the tab regains visibility (the load-bearing trigger, since the main interaction is "sent a link on Telegram → flip to dashboard"). All polling **pauses when the tab is hidden**. `/api/jobs/stats` is filter-independent and fetched **once on mount** (never per filter). Guardrail: the client-side model holds to ~1000 jobs (the realistic 1-year ceiling at ~175/3wk); above that, fall back to server-side filtering. Backend steady-state is ~0.25 s (Cloudflare-tunnel) ≈ ~0.23 s (localhost) — the ~13 s seen in profiling is a **cold-start** spike on the self-hosted side (idle tunnel/container), **not** query latency (175 rows, `idx_jobs_chat_id`, one Redis GET). So the backend "fix" is a **keep-warm ping** (`/health` every ~3–5 min), not query tuning; and the polling here keeps the backend warm while the dashboard is open, so cold start only ever bites the first load after a long idle. The ~2 min interval is a **discovery-urgency** choice, not a cost constraint. Retries are **not** part of this — they belong to the [[Recovery panel]]. |

| **Ops bot**                      | The internal Telegram operations surface for the [[Operator]], currently `TTTask_bot` / `@photo2urlBot`. The two bots use separate credentials and webhook validation: Ownix uses `TELEGRAM_BOT_TOKEN` / `TELEGRAM_WEBHOOK_SECRET` on `/webhook`, while the Ops bot uses `OPS_BOT_TOKEN` / `OPS_WEBHOOK_SECRET` on `/webhook/ops`. Access is split by explicit allowlists: `OPS_CHAT_IDS` may receive alerts and use read-only commands (`/pending`, `/users`, `/status`), while `OPS_ADMIN_CHAT_IDS` may perform mutating actions such as approving/blocking users, batch-approving pending users, triggering deploy actions, retrying failed jobs, or clearing queues. Invite approval/block keyboards are sent only to `OPS_ADMIN_CHAT_IDS`; read-only ops users may inspect pending users but do not receive decision buttons. When an admin approves or blocks a user from the Ops bot, the Ops bot confirms the internal action, while the user-facing Ownix bot sends any external outcome message to the affected user (for example, an approval notice with the app link); the Ops bot never becomes visible to end users. The first implementation slice includes invite approval notifications, `/pending`, `/users`, and a domain-scoped batch approval command (`/approve_pending <email-domain>`) that previews the count/list and requires an inline confirmation before changing statuses; there is no naked approve-all command in production. `/users` defaults to the 20 most recent users, `/users pending|approved|blocked` filters by status, `/users email <domain>` lists captured emails for a specific domain, and `/users email all` lists every captured email regardless of domain so custom-domain users are not missed. User-list results render directly in chat for 20 or fewer rows; larger result sets are sent as a CSV document. Broader production alerts and deploy notifications follow after the bot split is proven. Both allowlists can initially contain only the Operator, but the split keeps "can see ops" separate from "can change production." _Avoid_: treating Ops bot messages as user-facing product interactions. |
| **Approved user**                | A `users` row with `status='approved'`. The three states are `pending` (default-deny initial), `approved`, and `blocked` (denied); the Operator's decision moves a `pending` row to `approved` **or** `blocked` — they are not a linear chain. `OPERATOR_CHAT_ID` is auto-approved; pre-ADR-0031 rows are grandfathered `approved`. Carries a one-time `email` captured once per `tg_id`, shared across the bot and dashboard surfaces (Telegram never exposes phone — vig never stores it). |
| **Session identity**             | The Telegram-derived identity of the signed-in dashboard user (`id`, `first_name`, `username`, `photo_url`), minted into the session at login and returned by `GET /api/auth/me`. Fetched **once** by [[Invite gate]] and exposed to children via a session-user context (`useSessionUser()`), never re-fetched downstream. _Avoid_: profile, account. |
| **Dev login**                    | Local-only dashboard auth shortcut enabled by `DEV_LOGIN_ENABLED=true`. It creates a fake pending [[Session identity]] for testing the invite flow without the Telegram Login Widget. By default it stays local and quiet: no real Ops bot notification is sent. Full end-to-end Ops bot testing is opt-in via explicit local configuration (for example `OPS_DEV_NOTIFICATIONS=true` plus Ops bot credentials/admin allowlist), and generated approval cards must be clearly marked as local/dev. Dev approval cards go to `OPS_DEV_CHAT_IDS`, falling back to `OPS_ADMIN_CHAT_IDS` when the dev-specific list is unset, so integration-test noise can be isolated from production admins. Local dev also keeps a backend-only approval fallback so pending-to-approved dashboard testing does not require exposing localhost to Telegram. For true local callback testing, use an explicit dev-e2e script to start ngrok, expose the local API, and override `OPS_WEBHOOK_URL` in the backend `.env.local` to the ngrok `/webhook/ops` URL; `uvicorn src.main:app --reload` must not open a public tunnel as a side effect. Production keeps `OPS_WEBHOOK_URL` pointed at the production API. |
| **Google connection**            | The signed-in [[Tenant]]'s OAuth link to Google powering [[Opt-in export]] (Drive/Sheets). Connected ⇔ a stored refresh token exists (`GET /api/google/status`); disconnecting (`POST /api/google/disconnect`, behind a `window.confirm`) revokes the token server-side, so reconnecting requires the full consent flow. It gates exports, never access — Telegram is the only login. _Avoid_: Google account, Google login. |
| **Account affordance**           | The persistent sidebar-footer surface rendering the [[Session identity]] (avatar + name) with the [[Google connection]] state inline: a simple-icons Google mark, "Connected to Google" in Google brand blue `#4285F4` (deliberate off-system token — NOT signal orange, NOT the status ramp), and a muted Disconnect/Connect action. Collapsed rail shows the avatar only, with a static brand-blue glow when connected and a tooltip carrying name + state. Shared state lives in one dashboard-layout provider (`{ connected, disconnect, refresh }`) so the sidebar and the Feed's connect nudge never desync. The Feed's "Connect Google" panel is a **disconnected-only nudge** that disappears once connected — not a second source of truth. OAuth return (`/?google=connected\|denied`) renders a one-time inline banner on the Feed page (no toast system exists), then strips the param via `router.replace`. _Avoid_: profile widget, user menu. |

```

→ worker dequeues → short_video.run
  → frames service → Gemini Vision analysis → links
  → Drive upload ({job_id}_short.md) → Sheets append → Telegram delivery (photo + links)
  → fire-and-forget brain.ingest_links
  → status = done                          [vision deliverable complete]
  → ALWAYS: transcript step (ADR-0020) — best-effort, independent of template
    → transcript service
       ├─ captions → text (free)
       ├─ caption-less + no template → transcription-only Gemini call (free→paid key)
       └─ caption-less + template    → transcript falls out of the fused audio-enrichment call
    → persist jobs.transcript + key_phrases
    → if template set (slash command): Gemini enrichment → Telegram delivery (template_analysis)
    → TAIL (always last): upload {job_id}_transcript.md to Drive → send transcript document
       (failure notes: sidecar error / Gemini-unavailable / "⚠️ I'm wordless")

/freestyle <url> (slash command — works for both short and long URLs)
→ create job row (pending, template=freestyle)
→ arm chat_state(awaiting_freestyle, job_id) → ForceReply: "What should Gemini focus on?"
[user types prompt]
→ store jobs.freestyle_prompt → clear chat_state → enqueue {"task":"video"}
→ same pipeline.run path; enrichment uses freestyle_prompt in place of template extra_instructions
```

## Key Invariants

1. **Worker is the single lock acquirer for both PRD slots.** The webhook never sets `prd_auto_status='generating'` or `prd_intent_status='generating'`. This prevents TOCTOU races between two PRD trigger paths.
2. **Photo pipeline is inline.** No DB job, no Redis queue. Results go directly to Telegram. Brain ingest is fire-and-forget.
3. **Brain ingest never blocks a user-facing response.** Always wrapped in `asyncio.create_task(...)`.
4. **Free→paid Gemini key fallback.** All Gemini call paths (text, vision, photo, embed) share a single fallback loop in `src/services/gemini._call_with_fallback`; keys are read from `settings`, not passed as parameters. Anthropic keys were never in scope. See ADR-0011.
5. **SQLite WAL mode.** The API and worker both open the same SQLite file; WAL allows concurrent readers. Only one writer at a time — the worker owns all write-heavy paths.
6. **Verbatim-grounded photo links only.** Every link returned by `gemini_photo` must include a quoted substring from the image containing the domain. Post-filter in `_filter_grounded_links` drops anything ungrounded or matching UI chrome patterns.
7. **Article pipeline has no Drive upload.** Markdown is sent as a Telegram document only. No `drive_url` is written for article jobs.
8. **Repo pipeline has no Drive upload either.** The `<owner>-<repo>.md` analysis is sent as a Telegram document only. README lives on GitHub; analysis lives in Sheets + Telegram. No `drive_url` is written for repo jobs.
9. **Repo pipeline uses GitHub REST API, never Jina Reader.** See ADR-0014. Article pipeline uses Jina; repo pipeline does not. The two pipelines have different content-fetch services because the source shapes differ (arbitrary web pages vs one well-known platform with native tree/manifest endpoints).
10. **Repo pipeline has no allowlist.** Every `github.com/<owner>/<repo>[/...]` URL routes through automatically; reserved-path blocklist guards GitHub product pages (`/pricing`, `/features`, etc.). The article allowlist exists because "is this an article?" is fuzzy across thousands of hosts — `github.com` has none of that fuzziness.
11. **Repo bundle cache (Redis, 7d TTL) is separate from the photo-pipeline metadata cache (Redis, 24h TTL).** Both keyed by `{owner}/{repo}` but under different namespaces (`github_repo_bundle:` vs `github_meta:`). `/force <repo-url>` deletes both. Photo pipeline and `/find` continue to read only `github_meta:` for fresh star counts.
8. **Orphaned jobs are reset to error at startup, never re-queued.** A worker crash leaves a job in `processing`/`enriching`. The boot-time `reap_stale_jobs()` reaper marks it `error`, increments `attempt`, and notifies the user (enrichment → retry button; video → resend link). The video pipeline is not idempotent (re-running re-uploads Drive + re-appends Sheets), so auto re-queue is deliberately avoided (ADR-0010).
12. **FK enforcement is ON for every runtime connection.** `PRAGMA foreign_keys=ON` is set in the `connection()` context manager, not just `init_db`'s startup connection — because SQLite defaults it OFF per-connection. This is load-bearing for the [[web-dashboard]] tables, whose `ON DELETE CASCADE` clauses (`space_urls`, `context_blobs` ← `spaces`; `job_annotations`, `job_tags` ← `jobs`; `job_tags` ← `tags`) silently do nothing without it. Pre-web tables had almost no FKs, so enforcement was previously absent without harm.
13. **One [[Graph node]] per normalized URL.** [[Brain ingest]] normalizes a link's URL (strip query/fragment/trailing slash) before the `WHERE url = ?` dedup lookup; a re-sighting bumps `seen_count`, it never creates a second node. This is load-bearing for the Obsidian Drive sync, which writes `{slugify(title)}.md` — two nodes sharing a URL/title would collide on the same Drive file. Re-occurrence is **not** time-windowed (no "new node after N days"). See ADR-0027.
